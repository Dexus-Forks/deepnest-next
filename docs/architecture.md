# deepnest-next — Architecture

> Generated by Winston (BMAD Architect) on 2026-04-25 from a deep scan of the brownfield codebase. This document captures observed (not aspirational) architecture and is the canonical reference for AI-agent context.

## 1. Executive Summary

`deepnest-next` is an Electron-based desktop application with a **two-tier compute topology**:

- **Tier A — UI / orchestration**: visible renderer running TypeScript services + Ractive templates, hosted by the Electron main process which provides lifecycle, file persistence and IPC.
- **Tier B — Headless workers**: hidden Electron `BrowserWindow`(s) running the placement engine + native NFP add-on. Coordinated by the main process via `ipcMain` channels.

The genetic algorithm lives in Tier A; each individual it generates is sent to Tier B for evaluation. Results return through the same IPC pipe and feed back into the GA's fitness ranking.

There is no server, no database, no persistent network state. All persistence is local JSON files. External network calls are limited to: (1) DXF→SVG conversion at `converter.deepnest.app`, (2) optional OAuth login flow, (3) periodic notification polling.

## 2. Process & Window Topology

```
┌──────────────────────────────────────────────────────────────────┐
│ Electron Main Process (main.js)                                  │
│ - app lifecycle, single-instance lock                            │
│ - createMainWindow / createBackgroundWindows / createNotification│
│ - ipcMain handlers: read-config / write-config / load-presets…   │
│ - notification-service.js (HTTP poll every 30 min)               │
│ - presets.js (file-backed preset CRUD)                           │
│ - global.NEST_DIRECTORY (tmp scratch)                            │
└────────────┬───────────────────────────────────┬─────────────────┘
             │ ipcMain.on / handle               │
   ┌─────────▼─────────┐                ┌────────▼────────┐
   │ Main Renderer     │                │ Background      │
   │ main/index.html   │  background-*  │ Renderer (×1)   │
   │ → build/ui/...    │ ◄────────────► │ main/background │
   │ Ractive UI +      │                │ + @deepnest/    │
   │ DeepNest GA       │                │   calculate-nfp │
   │ (deepnest.js)     │                │   (native)      │
   └───────────────────┘                └─────────────────┘
             │
   ┌─────────▼─────────┐
   │ Notification      │  modal child window, opt-in marked-rendered
   │ main/notification │
   └───────────────────┘
```

**Counts**: exactly one main window, currently exactly one background window (`createBackgroundWindows()` short-circuits at `winCount < 1`; comment notes the legacy was 8). Notification window is created on demand and destroyed on close.

**Security posture (current, intentional)**:
- `contextIsolation: false`
- `nodeIntegration: true`
- `enableRemoteModule: true` via `@electron/remote`
- `nativeWindowOpen: true`

These let the renderer `require()` Node modules directly. Any hardening must change every `BrowserWindow` constructor in `main.js`.

## 3. Module Boundaries

### 3.1 Main process (privileged Node)

| Module | Responsibility | Key exports / IPC |
|---|---|---|
| `main.js` | App lifecycle, window creation, IPC fan-out | `ipcMain.on('background-*')`, `ipcMain.handle('read-config'\|'write-config'\|'load-presets'\|'save-preset'\|'delete-preset')` |
| `presets.js` | Read/write `presets.json` in `app.getPath('userData')` | `loadPresets`, `savePreset`, `deletePreset` |
| `notification-service.js` | Pull-based notification fetch + dedupe | `checkForNotifications()` |

### 3.2 UI renderer (`main/ui/`)

Modular TypeScript replacing the legacy `page.js`. Bootstrapped by `main/ui/index.ts` which wires services and components together.

```
main/ui/
├── index.ts                      # Composition root for renderer
├── types/index.ts                # UI-extended types, IPC channel constants
├── services/
│   ├── config.service.ts         # window.config (getSync/setSync/reset) → IPC
│   ├── preset.service.ts         # Preset CRUD → IPC (load/save/delete)
│   ├── import.service.ts         # SVG/DXF import dialog + parse pipeline
│   ├── export.service.ts         # SVG/DXF/JSON export
│   └── nesting.service.ts        # Drives DeepNest start/stop, dispatches to background
├── components/
│   ├── navigation.ts             # Tab navigation + dark-mode
│   ├── parts-view.ts             # Ractive-bound parts table
│   ├── nest-view.ts              # Ractive-bound nest result viewer
│   └── sheet-dialog.ts           # Add-sheet modal
└── utils/
    ├── dom-utils.ts              # getElement / getElements helpers
    ├── conversion.ts             # mm/inch + scale conversion
    └── ui-helpers.ts             # message(), small DOM helpers
```

### 3.3 Core engine (legacy CommonJS, JS)

| File | Lines | Role |
|---|---:|---|
| `main/deepnest.js` | 1658 | `DeepNest` class — owns `parts`, `imports`, `partsTree`, `nests`, `working`. Hosts the genetic algorithm wrapper. Calls `window.SvgParser.load`, dispatches placements via IPC `background-start`. |
| `main/background.js` | 2710 | Greedy NFP-based placement engine, hole-fitting, fitness scoring. Imports `NfpCache` and `HullPolygon` from compiled `build/`. Loads `@deepnest/calculate-nfp` native addon. |
| `main/svgparser.js` | 1577 | SVG load → polygon-tree extraction (paths, polylines, polygons, rect, ellipse). Closed-path detection, polygonification, scaling. |
| `main/nfpDb.ts` | 68 | `NfpCache` — in-memory keyed cache: `{A,B,Arot,Brot,Aflip,Bflip}` → `Nfp \| Nfp[]`. Defensive deep-clone on read/write. |

### 3.4 Vendored / legacy utilities (`main/util/`)

Treat as third-party libraries — do not modify casually:

- `clipper.js` — Vatti polygon clipping (ported)
- `simplify.js` — Ramer-Douglas-Peucker
- `ractive.js` — Ractive 0.x bundle
- `interact.js` — `interact.js` for resizable parts-list
- `parallel.js` — legacy worker pool (mostly unused in current build, replaced by IPC fan-out)
- `pathsegpolyfill.js` — SVGPathElement polyfill for Chromium
- `svgpanzoom.js` — SVG pan/zoom for import preview

Modernised TypeScript ports next to the JS originals:

- `point.ts`, `vector.ts`, `matrix.ts`, `HullPolygon.ts`, `domparser.ts`, `eval.ts`, `geometryutil.js` (still JS)

### 3.5 Native add-ons (out-of-tree)

- `@deepnest/calculate-nfp` — C/C++ Node addon doing the heavy NFP / Minkowski math. Rebuilt against the active Electron ABI by `electron-builder install-app-deps` (run automatically via `postinstall` and `npm run build`).
- `@deepnest/svg-preprocessor` — Used in the UI for `loadSvgString` and `simplifyPolygon`. Native module per repo conventions.

## 4. Data Model (in-memory)

Defined in `index.d.ts` (canonical) and `main/ui/types/index.ts` (UI extensions).

- `DeepNestConfig` — algorithm + UX configuration (units, scale, spacing, rotations, populationSize, mutationRate, placementType, mergeLines, timeRatio, simplify, dxfImport/Export scale, endpointTolerance, conversionServer).
- `UIConfig extends DeepNestConfig` — adds OAuth tokens, SVG preprocessor toggle, quantity-from-filename, sheet boundary export options.
- `Polygon extends Array<PolygonPoint>` — `id`, `source`, `children` (holes/nested), `parent`, `filename`.
- `Part` — `polygontree`, `svgelements`, `bounds`, `area`, `quantity`, `filename`, `sheet?`, `selected?`.
- `NestingResult` — `area`, `fitness`, `placements[]` (each: `sheet`, `sheetid`, `sheetplacements: SheetPlacement[]`).

Globals exposed on `window`:

- `window.config: ConfigObject`
- `window.DeepNest: DeepNestInstance`
- `window.SvgParser: SvgParserInstance`
- `window.nest: RactiveInstance` (nest-view Ractive)

These are explicit globals — see ADR-005.

## 5. IPC Contract

All channels are between main process and one renderer at a time.

### 5.1 Background nesting (renderer → main → background renderer)

| Channel | Direction | Payload | Notes |
|---|---|---|---|
| `background-start` | renderer → main → bg | `{individual, sheets, ids, sources, children, filenames, sheetids, sheetsources, sheetchildren, config}` | Main forwards to first idle bg window (`isBusy=false`). |
| `background-response` | bg → main → renderer | placement result | Main matches sender to its window slot, clears `isBusy`. |
| `background-progress` | bg → main → renderer | `{index, progress}` (negative = done) | Wrapped in try/catch — known `// todo:` from cmidgley fork: should cleanly stop workers on close. |
| `background-stop` | renderer → main | none | Destroys all bg windows and recreates them. |

### 5.2 Configuration & presets

| Channel | Kind | Payload | Persistence |
|---|---|---|---|
| `read-config` | `handle` | — → `object` | Reads `userData/settings.json`, replaces legacy `convert.deepnest.io` URLs with `converter.deepnest.app`. |
| `write-config` | `handle` | `string` (stringified) | Writes `userData/settings.json` verbatim. |
| `load-presets` | `handle` | — → `object` | `userData/presets.json` (same URL rewrite). |
| `save-preset` | `handle` | `name, config` | Merge & write. |
| `delete-preset` | `handle` | `name` | Remove key & write. |

### 5.3 Auxiliary

| Channel | Kind | Purpose |
|---|---|---|
| `login-success`, `purchase-success` | `on` (renderer broadcasts) | Forwarded to main window for cloud feature UI updates. |
| `setPlacements`, `test` | `on` | Stash payload on `global.exportedPlacements` / `global.test` for remote inspection. |
| `get-notification-data`, `close-notification` | notification window only | Read content; mark as seen + advance to next. |

## 6. Build & Compile Topology

```
┌────────────────────────────────────────────────────────────┐
│ npm run build                                              │
│  1. tsc                                                    │
│     - input:  index.d.ts + main/**/*.ts (per tsconfig)     │
│     - output: ./build/ (preserves directory layout)        │
│  2. electron-builder install-app-deps                      │
│     - rebuilds @deepnest/calculate-nfp etc. against        │
│       electron@^40 ABI                                     │
└────────────────────────────────────────────────────────────┘
```

`main/index.html` and `main/background.html` reference compiled output via relative paths into `build/`. Legacy `.js` files in `main/` (deepnest.js, background.js, svgparser.js, util/*.js) are loaded directly without compilation.

The mixing of compiled and uncompiled JS is intentional — the legacy core engine has not been ported yet. Anything new should be TypeScript and live under `main/ui/` or a sibling typed module.

## 7. Cross-Cutting Concerns

- **Configuration**: `ConfigService.getSync` / `setSync` is the only sanctioned read/write path. Defaults live in `services/config.service.ts`. `BOOLEAN_CONFIG_KEYS` exported as the source-of-truth list of checkbox-bound keys.
- **Units**: Internal store is **inches** (scale value). `mm` is presented in UI by dividing by 25.4 in `updateForm()` and multiplying back on change. Don't double-convert — see the explicit guards in `initializeConfigForm()`.
- **Coordinate scaling**: Two scales coexist — `config.scale` (SVG units) and `config.clipperScale` (1e7) for Clipper integer arithmetic.
- **Caching**: `NfpCache` in `main/nfpDb.ts` lives per background renderer. Scratch directory `os.tmpdir()/nest` is wiped on `app.before-quit`.
- **Fonts/Assets**: Two Lato webfont families bundled under `main/font/`. Icons in `main/img/` and `icon.icns`/`icon.ico` at root.
- **Logging**: Plain `console.*`. Crash reports via `crashReporter.start({uploadToServer:false})`.
- **Listener cap**: `EventEmitter.defaultMaxListeners = 30` (set in `main.js`).

## 10. Architectural Decision Records

> ADRs capture decisions whose rationale is non-obvious from the code alone. They are observations of the existing system, not new proposals — when you change one of these, supersede the ADR.

### ADR-001 — Multi-process compute via hidden BrowserWindows

**Status**: Accepted (inherited).

**Context**: Nesting is CPU-bound and blocks the UI. Node `worker_threads` cannot host the native NFP addon directly because it depends on Electron-specific globals (`window`, `ipcRenderer`).

**Decision**: Run the placement engine in a hidden `BrowserWindow` (Tier B) and coordinate via `ipcMain`.

**Consequences**:
- (+) Worker keeps full Electron ABI compatibility for the native addon.
- (+) Each background window is isolated; crashes log to `render-process-gone` and don't kill the main UI.
- (−) Higher memory footprint per worker (~one Chromium renderer each).
- (−) IPC payload duplication — large polygon trees are stringified through structured-clone.

**Note**: Worker count was historically 8, currently hard-coded to 1 in `createBackgroundWindows`. Increasing this requires re-checking `background-start` round-robin logic.

### ADR-002 — Genetic algorithm in main renderer, placement in background

**Status**: Accepted (inherited).

**Context**: GA needs visibility of `parts`, UI selection state, and progress callbacks. Placement is heavy and must offload.

**Decision**: GA orchestrator (`DeepNest.start()` in `main/deepnest.js`) lives in the visible renderer; each `individual` is shipped to the background renderer for evaluation; the result feeds back into the GA population in the visible renderer.

**Consequences**:
- (+) UI can render intermediate placements as soon as `background-response` arrives.
- (−) Visible renderer cannot be fully decoupled from compute — heavy DOM updates compete with GA tick.

### ADR-003 — Native NFP / Minkowski as out-of-tree npm packages

**Status**: Accepted (inherited).

**Context**: Original repo embedded a `binding.gyp` and C/C++ source. cmidgley/deepnest-next moved this out so the desktop repo focuses on UI/orchestration.

**Decision**: NFP/Minkowski compute lives in `@deepnest/calculate-nfp` (and the upcoming Rust replacements per README). SVG preprocessing lives in `@deepnest/svg-preprocessor`.

**Consequences**:
- (+) Native build complexity is contained.
- (+) Rust migration can land per-package without touching this repo.
- (−) Local development requires rebuilding native deps after every Electron major bump (`electron-builder install-app-deps` is wired into `postinstall`).
- (−) Source-level debugging of NFP requires checking out the addon repo separately.

### ADR-004 — `nodeIntegration: true` + `contextIsolation: false`

**Status**: Accepted with risk.

**Context**: Renderer uses `require('electron')`, `require('graceful-fs')`, `require('@deepnest/svg-preprocessor')`, etc. Migrating to a preload-script bridge would be a substantial refactor.

**Decision**: Keep current renderer config; deepnest does not load remote content, mitigating the standard XSS-to-RCE risk.

**Consequences**:
- (+) Direct Node access keeps the legacy code paths working.
- (−) Any future feature that loads untrusted content (e.g. user-supplied HTML in notifications) must be sandboxed.
- (−) Electron security audits will flag this. Document explicitly in any security review.

**Compensating control**: `mainWindow.webContents.setWindowOpenHandler(...)` rejects all `window.open` and routes URLs to the OS shell.

### ADR-005 — Globals on `window` (DeepNest, SvgParser, config, nest)

**Status**: Accepted (inherited).

**Context**: Pre-TypeScript code attached singletons to `window`. The new modular UI must talk to them.

**Decision**: Declare these as ambient globals in `index.d.ts` and access via getter helpers (`getDeepNest()`, `getSvgParser()`) in `main/ui/index.ts`. Do not re-introduce additional globals.

**Consequences**:
- (+) Bridge between legacy and new code is explicit and typed.
- (−) Unit tests can't easily stub these — Playwright E2E only.

### ADR-006 — Ractive.js for parts-view & nest-view

**Status**: Accepted (legacy).

**Context**: Original Deepnest UI used Ractive 0.x. Replacing it with React/Vue would invalidate every template attribute and binding.

**Decision**: Keep Ractive for the two main data tables (`parts-view`, `nest-view`). New UI work should prefer plain DOM + the existing service layer rather than expanding Ractive usage.

**Consequences**:
- (+) Stable, no rewrite cost.
- (−) Two UI paradigms coexist — contributors must read both.

### ADR-007 — Strict TypeScript only on `main/**/*.ts`

**Status**: Accepted.

**Context**: `tsconfig.json` `include` is `main/**/*.ts` and `main/ui/**/*.ts`. The compiler does not touch `main.js`, `main/deepnest.js`, `main/background.js` or `main/util/*.js`.

**Decision**: Strict mode (`strict`, `noImplicitAny`, `strictNullChecks`, `noUnusedLocals`, `noUnusedParameters`, `noImplicitReturns`, `noFallthroughCasesInSwitch`) for typed code. Legacy JS stays as-is until per-module ports.

**Consequences**:
- (+) New modules cannot regress on type safety.
- (−) Type information for the legacy modules lives entirely in `index.d.ts` and `main/ui/types/index.ts` — keep these in sync when the JS changes.

> **Note on ADR-008 / ADR-009**: these numbers are reserved by the MVP planning overlay (`_bmad-output/planning-artifacts/architecture.md` §5 — ADR-008 LICENSES generator, ADR-009 background-worker shutdown sentinel). They are not yet promoted into this canonical file. ADR-010 below skips that gap deliberately — when 008/009 are promoted, they slot in above.

### ADR-010 — Webfonts ship `woff2` + `woff` only

**Status**: Accepted.

**Context**: The bundled Lato webfont kit historically shipped four `src:` formats per face: `eot`, `woff2`, `woff`, `ttf`. `eot` is an IE-only legacy format (introduced for IE9 and earlier); `ttf` was a secondary fallback for very old browsers. Story 1.1 (DEE-55, MVP-1 Stream A1) removed the `.eot` and `.ttf` files for the three live LatoLatin weights (Bold, Regular, Light) and trimmed the corresponding `src:` URLs from `main/font/latolatinfonts.css`. Measured installer-size saving for this slice: ~650 KB across 6 files (full Story-1.1 footprint reduction was ~1.85 MB, see `docs/asset-inventory.md` §6 P4 row + §8 verification recap). This ADR codifies the post-Story-1.1 state as a binding architectural rule.

**Decision**: Webfonts in this app ship as a `{woff2, woff}` pair only. `.eot` and `.ttf` `src:` URLs MUST NOT be reintroduced in `main/font/latolatinfonts.css` or any sibling stylesheet under `main/`. Adding a new webfont follows the same `{woff2, woff}` set: declare via `@font-face` in `main/font/`, listing both `format("woff2")` and `format("woff")` `src:` entries.

**Consequences**:
- (+) Smaller installer footprint per added face — no dead-weight legacy formats ship.
- (+) Single canonical loading path; reviewers can grep `src:` URLs against this ADR mechanically.
- (+) Removes the implicit-fallback chain that previously masked rendering regressions on the live Chromium-only target.
- (−) The rule is justified only by the Chromium-only deployment context (ADR-001 + ADR-004). If a future story adds a non-Electron renderer surface (e.g. a web build, or a renderer that loads remote content), this ADR must be revisited per face — `.woff` is still the appropriate broad-compatibility companion to `.woff2`, but `.ttf` may legitimately re-enter the matrix in that hypothetical.
- (−) `LatoLatin-BoldItalic.{woff,woff2}` stays out of scope — it had no `@font-face` binding pre-Story-1.1 (P1 in `docs/asset-inventory.md` §6) and is not reintroduced by this ADR.

**Cross-references**:
- ADR-001 (multi-process Electron), ADR-004 (`nodeIntegration: true`, no remote content) — establish the Chromium-only deployment context that makes `.eot`/`.ttf` redundant.
- `_bmad-output/project-context.md` §9.9 — invariant binding for code-touching agents (canonical wording).
- `docs/asset-inventory.md` §4 (cross-link), §6 P4 (measured saving), §8 (verification recap).
- Story 1.1 / DEE-55 (PR #8, merge `8c3afa8`) — origin of the removal.

## 13. Glossary

- **NFP** — No-Fit Polygon. Geometric locus of all positions where one polygon would touch but not overlap another.
- **Inner NFP** — set of valid placements inside a container (sheet or hole).
- **Outer NFP** — forbidden positions around an already-placed part.
- **Hole-fitting** — packing smaller parts into the cutouts of larger parts.
- **Fitness** (in this codebase) — lower is better; the GA minimises it.
- **Sheet** — a part with `sheet: true`; treated as a bin instead of a piece.
- **Polygontree** — outer polygon with nested `children` arrays representing holes / nested parts.
