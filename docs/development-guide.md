## Prerequisites

| Tool | Required version | Notes |
|---|---|---|
| Node.js | `20.x` or `22.x` | Matrix-tested in CI. Use `nvm` / `nvm-windows`. |
| npm | bundled with Node | `npm install` is canonical (no Yarn/pnpm support). |
| Python | `3.7.9+` | Required by `node-gyp` for native rebuilds. On Windows 10 1905+, disable the App Installer Python alias. |
| Rust toolchain | latest stable (rustup) | Currently for upcoming `@deepnest/*` Rust modules â€” **not** required to build this repo today. |

### Per-platform extras

- **Windows**: Visual Studio 2022 with the *Desktop development with C++* workload. Older alternative: `npm install --global windows-build-tools` (admin shell).
- **macOS**: `xcode-select --install` then `brew install boost`.
- **Linux**: `gcc`, `clang`, `libboost-dev` (CI uses `apt-get install -yq libboost-dev`).

> Boost is required by the upstream native modules (`@deepnest/calculate-nfp`); it isn't built locally inside this repo, but `electron-builder install-app-deps` triggers the addon build during `postinstall`.

## First-time setup

```sh
git clone https://github.com/deepnest-next/deepnest
cd deepnest
npm install         # also runs `electron-builder install-app-deps` via postinstall
npm run build       # tsc + native deps rebuild
npm run start       # launches Electron with the built app
```

`npm install` will rebuild `@deepnest/calculate-nfp` and `@deepnest/svg-preprocessor` against the active Electron ABI. Re-run `npm install` (or `npm run build`) after upgrading Electron.

## Day-to-day commands

| Command | What it does |
|---|---|
| `npm run start` | `electron .` â€” launches the app from current sources (uses compiled `build/` for TS files). |
| `npm run build` | `tsc && electron-builder install-app-deps`. Compiles `main/**/*.ts` and rebuilds native deps. |
| `npm run clean` | Removes `build/` and `dist/` (Windows-only `rmdir`; on Linux/macOS use `rm -rf build dist`). |
| `npm run clean-all` | Also wipes `node_modules` and `bin`. |
| `npm test` | Runs the Playwright E2E suite (`tests/index.spec.ts`). |
| `npm run pw:codegen` | Launches Playwright codegen against the running Electron app (`helper_scripts/playwright_codegen.js`). |
| `npm run dist` | `@electron/packager` build â†’ `./deepnest-v${version}-win32-${arch}/`. |
| `npm run build-dist` | `electron-builder build --win` (signed). |
| `npm run dist-all` | Full clean + install + build + dist. |

## Editing TypeScript vs. JavaScript

- New code â†’ TypeScript under `main/ui/` (or a sibling typed location).
- Editing `main.js`, `main/deepnest.js`, `main/background.js`, `main/svgparser.js`, `main/util/*.js` â†’ permitted, but bear in mind:
  - These files are loaded **directly** by the renderer/main â€” there is no transpile step. Stay on Node-supported ES syntax (no TS, no JSX).
  - Type information for these modules lives in `index.d.ts` and `main/ui/types/index.ts`. Update them when you change exports / globals.
  - No unit tests cover them â€” verify behaviour via `npm test` (Playwright) or manual run.

## Testing

### Playwright E2E

```sh
npx playwright install chromium    # one-time
npm test
```

Configuration highlights (`playwright.config.ts`):
- `headless: false` (Electron app is launched)
- `video: 'on'` locally, `'retain-on-failure'` in CI
- `screenshot: 'on'`, `trace: 'on-first-retry'`
- `forbidOnly` + `retries: 2` + `workers: 1` in CI
- `projects: [{ name: 'chromium' }]` only

Test fixtures under `tests/assets/`. The single spec is `tests/index.spec.ts`.

### Adding tests

```sh
node ./helper_scripts/playwright_codegen.js     # cross-platform
# or
npm run pw:codegen
```

Copy the recorded test into `tests/` and prune the boilerplate.

### What's not tested

- Anything below the UI: NFP cache, SVG parsing, GA fitness scoring, IPC payload shapes. These are reachable only via E2E today. Treat as a coverage debt.

## Debugging

### Open Chromium DevTools on every renderer

```sh
deepnest_debug=1 npm run start
```

This sets `process.env.deepnest_debug` which `main.js` reads in `createMainWindow`, `createBackgroundWindows`, and `createNotificationWindow`.

### Inspect the main process

Add `--inspect=9229` to the `electron .` invocation, then attach Chrome DevTools to `chrome://inspect`. (Not currently scripted in `package.json`.)

### Override scratch directory

```sh
SAVE_PLACEMENTS_PATH=/some/dir npm run start
```

Replaces the default `os.tmpdir()/nest`.

### Crash reports

`crashReporter.start({ uploadToServer: false })` is enabled in `main.js`. The most recent crash file path is logged on startup via `crashReporter.getLastCrashReport()`.

## Common Workflows

### Adding a new config option

1. Extend `DeepNestConfig` in `index.d.ts` and `UIConfig` in `main/ui/types/index.ts` if UI-only.
2. Add a default value in `main/ui/services/config.service.ts`.
3. Add the form input to `main/index.html` with `data-config="<key>"` (and `data-conversion="true"` if it's a length value).
4. If it's a checkbox, add the key to `BOOLEAN_CONFIG_KEYS` in `config.service.ts`.
5. Verify by toggling in the UI and reopening â€” `settings.json` in `app.getPath('userData')` should round-trip the value.

### Adding a new IPC channel

1. Decide who owns persistence (main process) vs. fan-out (renderer â†” background).
2. Pick a channel name; add it to `IPC_CHANNELS` constants in `main/ui/types/index.ts`.
3. In `main.js`, register `ipcMain.on(...)` (fire-and-forget) or `ipcMain.handle(...)` (request/response).
4. In the renderer service, call `ipcRenderer.send(...)` or `await ipcRenderer.invoke(...)`.
5. For background-fanout channels, follow the `background-start` / `background-response` pattern in `main.js`.

### Adding a new export format

1. Implement the producer in `main/ui/services/export.service.ts`.
2. Add a button in `main/index.html` with a unique id.
3. Bind it in `initializeExportButtons()` in `main/ui/index.ts`.

## CI Expectations

The build job (`.github/workflows/build.yml`) runs on every push to `main` and every PR:

- Matrix: `node 20.x | 22.x` Ă— `windows-2022 | ubuntu-22.04` (no macOS runner).
- Steps: checkout (with submodules) â†’ `apt install libboost-dev` (Linux) â†’ `npm install` â†’ `npm run build` â†’ `npm run dist`.

Playwright tests run via the separate `playwright.yml` workflow.

A failing `npm run build` on Linux is almost always a missing `libboost-dev` or a Python alias collision on Windows runners â€” recheck environment before debugging code.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `MODULE_NOT_FOUND: '@deepnest/calculate-nfp'` at runtime | Native rebuild skipped after Electron upgrade | `npm run build` (re-runs `electron-builder install-app-deps`) |
| Background nesting hangs forever | Background renderer crashed silently | Set `deepnest_debug=1`, inspect bg renderer console; check `render-process-gone` logs |
| Playwright timing out at 30s | Chromium not installed in `~/.cache/ms-playwright` | `npx playwright install chromium` |
| App fails to start with "second instance" loop | Stale single-instance lock | Kill remaining Electron processes; relaunch |
| Shell-quoted scripts fail on Windows | The `clean*` scripts use `rmdir /s /q` | Run on Windows shell, or replace with `rm -rf` for local hacks |
| `nfpcache/` directory not cleaned on quit | App killed before `before-quit` fired | Manually delete `os.tmpdir()/nfpcache` |
