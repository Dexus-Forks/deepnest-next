# deepnest-next — Project Overview

> Generated by Winston (BMAD Architect) on 2026-04-25 as part of DEE-10 brownfield documentation.

## Purpose

`deepnest-next` is a fast, open-source 2D nesting application for plotters, laser cutters, and other CNC tools. It packs irregular SVG/DXF shapes onto stock sheets while minimising waste and (optionally) cut-path length. The project descends from `SVGnest` → `Deepnest` (Jack000) → `cmidgley/Deepnest` → the present community fork (`deepnest-next/deepnest`).

## Repository Classification

- **Type**: Desktop application (Electron)
- **`project_type_id`**: `desktop`
- **Topology**: Monolithic Electron app with three logical surfaces
  1. **Main process** (`main.js`, `presets.js`, `notification-service.js`) — privileged Node, lifecycle, IPC orchestration, file persistence
  2. **Renderer (UI)** (`main/index.html` + compiled `main/ui/`) — TypeScript front-end built on Ractive.js + custom services
  3. **Background renderer(s)** (`main/background.html` + `main/background.js`) — hidden `BrowserWindow` running CPU-bound nesting/NFP code via native add-ons
- **No HTTP server, no database** — all state is in-memory with electron-settings-style JSON files (`settings.json`, `presets.json`) under the OS user-data directory.
- External web service: `https://converter.deepnest.app/convert` for DXF→SVG conversion.

## High-Level Capabilities

- Import SVG (and DXF via remote conversion) — extracts polygon trees through `main/svgparser.js`
- Genetic-algorithm orchestrated nesting (`main/deepnest.js`) over NFP-based greedy placement (`main/background.js`)
- Native NFP/Minkowski computation through `@deepnest/calculate-nfp` (external Node native module)
- SVG cleaning/preprocessing through `@deepnest/svg-preprocessor`
- Sheet management (rect / custom polygon sheets), part quantities, rotation count, mutation rate, etc.
- Export: SVG, DXF (via remote converter), JSON
- Preset management (save/load/delete named configurations)
- In-app notifications fetched from a remote channel (`notification-service.js`)
- OAuth login surface for cloud features (`access_token`/`id_token` carried through config)

## Architecture Pattern

**Multi-process Electron with worker offloading.** The main process owns lifecycle and IPC; the visible renderer drives the UI; one or more hidden renderers run the nesting algorithm so the UI thread stays responsive. State flows through `ipcMain.on('background-start' | 'background-response' | 'background-progress' | 'background-stop')` channels.

The nesting workload itself is hybrid:
- **Outer loop** (genetic algorithm) lives in the main UI renderer (`DeepNest` class in `main/deepnest.js`) — generates candidate orderings/rotations.
- **Inner loop** (placement + NFP) runs in the background renderer (`main/background.js`) which delegates the heaviest geometry to the native add-on.

See `architecture.md` for the full breakdown and ADRs.

## Brownfield Caveats

This codebase carries roughly a decade of fork history. A few practical notes for new contributors and AI agents:

- **Mixed module systems**: `main.js`, `main/deepnest.js`, `main/background.js`, and several `main/util/*.js` files are CommonJS. The newer `main/ui/**` is ES modules (TypeScript output). Build scripts mix the two via Electron's `nodeIntegration: true` and explicit `require()` calls.
- **Mixed style**: ESLint/Prettier enforcement only covers files staged via `lint-staged`. Older files (e.g. `main/util/clipper.js`, `simplify.js`, `pathsegpolyfill.js`, `parallel.js`) are kept verbatim from upstream and should be treated as vendored libraries.
- **Active rewrites**: README announces ongoing migration of compute-heavy code from C/C++ to Rust modules. Architecture decisions for those modules currently live outside this repo (in the `@deepnest/*` packages).
- **`webPreferences.contextIsolation: false` and `nodeIntegration: true`** are set across all windows — this is how the renderer reaches `require('electron')`, `graceful-fs`, etc. Any future Electron security hardening must touch every renderer entry point.
