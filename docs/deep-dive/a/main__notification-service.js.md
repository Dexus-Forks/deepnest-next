# Deep-Dive: `notification-service.js`

> Group A ¬∑ privileged Node module. Generated by Paige (Tech Writer) on 2026-04-25 from `notification-service.js@HEAD`.
>

The doc filename `main__notification-service.js.md` follows the convention prescribed by the issue scope, not the on-disk location. See ¬ß11.

## 1. Purpose

A pull-based notification fetcher with three responsibilities:

1. Discover new GitHub releases for `deepnest-next/deepnest`.
2. Fetch curated app notifications from `https://www.deepnest.net/app_notifications.json`.
3. Track which notifications the user has already seen and surface only unseen ones, optionally merging multiple "important"/"update" notifications into a single window.

It is **stateful**: it owns a JSON file under `userData` (`seen-notifications.json`) and persists every dismissal back to disk. It does not push ‚Äî `main.js` calls `checkForNotifications()` on a timer.

## 2. Public surface

```js
class NotificationService {
  constructor()            // synchronous: loads + cleans seen data
  async checkForNotifications(): Promise<NotificationPayload | null>
  // private (called from inside the class):
  loadSeenData()
  saveSeenData()
  cleanupOldNotifications()
  getLatestGithubRelease(): Promise<GithubRelease>
  getAppNotifications(): Promise<{ notifications: AppNotification[] } | { notifications: [] }>
  formatReleaseContent(release): string  // HTML
  formatFileSize(bytes): string
}
module.exports = NotificationService;
```

The single sanctioned consumer is `main.js`:

- `const notificationService = new NotificationService();` (`main.js:66`)
- `await notificationService.checkForNotifications();` (`main.js:206, 414`)

The notification window itself never imports this module ‚Äî it receives the resolved `{ title, content, markAsSeen }` payload via the `notification.notificationData` window-instance stash.

## 3. Polling contract

Defined entirely in `main.js`, not here. Three triggers exist:

| Trigger | Source line | Cadence |
|---|---|---|
| Initial check after first show | `main.js:263-265` | one-shot, 3 s after `mainWindow` becomes ready |
| Recurring background poll | `main.js:267-269` | `setInterval(... 30 * 60 * 1000)` ‚Äî every 30 min |
| Chained check after dismissal | `main.js:413-418` | one-shot, 500 ms after the user clicks OK |

`runNotificationCheck()` (`main.js:205-210`) is the only wrapper. It awaits `checkForNotifications()` and, if the result is non-null, opens a notification window.

There is no exponential back-off, no jitter, and no detection of wake-from-sleep. On macOS App Nap the 30-min interval drifts; on a laptop suspended for hours the next check runs as soon as the timer fires after wake.

## 4. Dedupe contract

### 4.1 Dedupe key

The dedupe key is the **notification UUID** (`AppNotification.uuid`). The on-disk shape is:

```json
{
  "lastSeenRelease": "v1.2.0",
  "seenNotificationIds": [
    { "uuid": "notification-2023-11-01-001", "viewedAt": "2023-11-05T14:32:45Z" }
  ]
}
```

Stored at `app.getPath('userData')/seen-notifications.json` (`notification-service.js:10`). Schema mirrors `examples/seen-notifications-example.json`.

### 4.2 What's checked

In `checkForNotifications()` (`notification-service.js:178-181`):

```js
const seenUuids = this.seenData.seenNotificationIds.map(item => item.uuid);
const unseenNotifications = allNotifications.filter(n => !seenUuids.includes(n.uuid));
```

UUID-only. The notification's `date` field is **not** part of the dedupe key ‚Äî re-publishing the same UUID with a new date does not re-surface it.

### 4.3 Migration shim

`loadSeenData()` (`notification-service.js:17-41`) detects the legacy format (array of plain UUID strings) and converts each entry into the `{ uuid, viewedAt: now }` shape on first read. The conversion uses the *current* time, not the original view time ‚Äî old installs lose precise view timestamps but keep their dedupe.

### 4.4 GitHub release dedupe

Separate from notification UUIDs. Tracked as a single `lastSeenRelease: string | null` (`notification-service.js:40`). A release counts as "new" iff (`notification-service.js:162-163`):

```js
releaseInfo.tag_name !== this.seenData.lastSeenRelease &&
releaseInfo.tag_name !== `v${this.appVersion}`
```

This skips the release that matches the currently-running version (so the user is not pestered to "upgrade" to what they already run) **and** the most recently dismissed release tag.

### 4.5 90-day cleanup

`cleanupOldNotifications()` (`notification-service.js:53-82`) is called from the constructor (`main.js`-time) **and** at the top of every `checkForNotifications()` (`notification-service.js:158`). It evicts entries whose `viewedAt` is older than 90 days. Side effect: a notification re-published with the same UUID after 90 days *will* re-surface ‚Äî by design, since its dedupe entry has been pruned.

> **Bug ‚Äî `cleanupOldNotifications` early-return is broken.** Line 54 reads `if (!this.seenData.seenNotificationIds || !Array.isArray(this.seenNotificationIds))`. The second clause references `this.seenNotificationIds` (no `seenData.`) which is always `undefined`, so the predicate is always truthy and the function returns early on every call ‚Äî pruning never runs. Recommend fixing to `!Array.isArray(this.seenData.seenNotificationIds)`. Filed for follow-up; the risk is unbounded growth of `seen-notifications.json` over years of use.

## 5. Schema returned to `main/notification.html`

Three result variants, all sharing the same exterior shape:

```ts
type NotificationPayload = {
  type: 'merged' | 'notification' | 'release';
  data:
    | { notifications: AppNotification[] }   // when type === 'merged'
    | AppNotification                        // when type === 'notification'
    | GithubRelease;                         // when type === 'release'
  title: string;                             // assigned to <h1 id="title">
  content: string;                           // HTML, assigned to #content via .innerHTML
  markAsSeen: () => void;                    // called by `close-notification` IPC handler
};
```

`notification.html:207-210` consumes only `{ title, content }` ‚Äî the `data` property is never read by the renderer; it exists for debug introspection from the main process. `markAsSeen` lives on the BrowserWindow's `notificationData` stash and is invoked from `main.js:403-405` (the `close-notification` handler).

### 5.1 Variant: merged important/update notifications (`notification-service.js:194-244`)

Triggered when `unseenNotifications` contains **more than one** notification of `type === 'important'` or `type === 'update'`.

- `title = "Multiple Important Updates"`.
- `content` is server-built HTML containing an `<h2>Important Updates</h2>` header, a sort note, and one `<div class="merged-notification">` per notification with title/date/content/HR.
- `markAsSeen` records every UUID in the bundle with the same `viewedAt` timestamp.
- If non-important/update notifications also exist, an extra `<div class="additional-notice">` is appended noting how many will appear after dismissal.

### 5.2 Variant: single important/update OR plain notification (`notification-service.js:245-290`)

Triggered when there is exactly one `important`/`update`, or zero of those plus any other type.

- `title = notification.title || 'Deepnest Notification'`.
- `content = notification.content` ‚Äî verbatim HTML from the upstream JSON.
- If there are additional non-important notifications behind it, the same `additional-notice` block is appended.
- `markAsSeen` records only the dismissed UUID.

### 5.3 Variant: GitHub release (`notification-service.js:292-304`)

Only surfaced when there is **no** unseen `AppNotification`. Falling through to release surfacing is intentional ‚Äî app notifications take priority.

- `title = "New Version Available: ${tag_name}"`.
- `content` built by `formatReleaseContent(release)` (`notification-service.js:314-357`):
  - `<h2>` with `release.name || release.tag_name`.
  - Publication date via `Intl.DateTimeFormat` (full date, long time, UTC).
  - `marked.parse(release.body)` of the markdown release notes.
  - `release.assets` ‚Üí `<ul class="assets-list">` of download links with `formatFileSize` (KiB-step) and `download_count`.
  - Trailing `<a href={release.html_url}>View on GitHub</a>`.
- `markAsSeen` stores `lastSeenRelease = release.tag_name`.

## 6. Upstream contracts

### 6.1 GitHub release

```
GET https://api.github.com/repos/deepnest-next/deepnest/releases/latest
Headers: User-Agent: deepnest-next-app/<package version>
```

Code: `getLatestGithubRelease()` (`notification-service.js:85-112`). Uses `https.get` (no timeout, no retry). Resolves with the parsed release JSON; rejects on non-200 or malformed JSON. **Caught and demoted to `null`** at the call site (`notification-service.js:161`) so an unreachable GitHub API never blocks the rest of the check.

### 6.2 App notifications

```
GET https://www.deepnest.net/app_notifications.json
```

Code: `getAppNotifications()` (`notification-service.js:115-152`). Same `https.get`, no timeout/retry. Rejection is caught and replaced with `{ notifications: [] }` at the call site (`notification-service.js:166`).

**Debug override**: when `process.env.deepnest_debug === "1"` and `examples/app_notifications.json` exists, the file is read instead of the network. Logs the override path. (`notification-service.js:117-132`.) This mirrors the env-var gate that opens DevTools in `main.js`.

Expected payload shape (verified against `examples/app_notifications.json`):

```json
{
  "notifications": [
    {
      "uuid": "notification-2023-11-01-001",
      "type": "important" | "update" | "preview" | "ads",
      "date": "2023-11-01T01:00:00Z",
      "title": "...",
      "content": "<h2>...</h2><p>...</p>"   // raw HTML, injected via innerHTML
    }
  ],
  "meta": { "lastUpdated": "...", "version": "..." }
}
```

The `meta` block is read but unused. The `type` value drives the merge/priority logic; only `important` and `update` trigger the merged window. `preview` and `ads` are surfaced through the single-notification path *only* when no important/update is pending.

The `content` field is **HTML and is injected via `innerHTML`** in `notification.html:209`. There is no sanitisation ‚Äî the upstream JSON is fully trusted. If `app_notifications.json` is ever served from a less-trusted origin, this becomes an XSS vector that runs with full Node integration (see ¬ß8).

## 7. Sort and priority

`checkForNotifications()` sorts all upstream notifications by parsed date, oldest first (`notification-service.js:170-175`):

```js
allNotifications = allNotifications
  .map(n => ({ ...n, parsedDate: n.date ? new Date(n.date) : new Date(0) }))
  .sort((a, b) => a.parsedDate - b.parsedDate);
```

Missing-date notifications sort to the very front. Within unseen notifications, the priority order presented to the user is:

1. **Merged important/update** (if 2+ exist).
2. **Single important/update** (if exactly one exists).
3. **Other types** (preview, ads, anything not important/update) ‚Äî first by date.
4. **GitHub release** (only if no `AppNotification` is pending).

Multi-step dismissal: the chained-check timer in `main.js:413-418` calls `checkForNotifications()` again 500 ms after dismissal, which surfaces the next-priority unseen item.

## 8. Security / risk notes

- **`content` is HTML, injected verbatim**. Both `notification.content` (for `AppNotification`) and `release.body` (rendered through `marked.parse`) end up inside the notification window's DOM via `.innerHTML`. The notification window runs with `nodeIntegration: true` (`main.js:167-171`), so any successful HTML injection has access to `require('electron')`. Mitigations today are entirely upstream: trust in `deepnest.net` and `api.github.com`. Hardening would require sanitisation here (e.g. DOMPurify) **and** a preload-bridge rewrite of `main.js`'s notification window so we can drop `nodeIntegration`.
- **`marked.parse` runs in the main process** (`notification-service.js:5, 350`). Marked supports HTML passthrough by default; a malicious release body cannot escape the renderer here, but it does construct arbitrary HTML that the renderer then mounts.
- **No request timeouts**. A dead network can leave `https.get` hanging until the OS TCP timeout (~minutes). The `await` at the call site blocks `runNotificationCheck`; the next 30-min tick keeps firing regardless.
- **`saveSeenData` is non-atomic** (`notification-service.js:46`). Same caveat as `presets.js` ‚Äî a crash mid-write truncates the file. Worst case: the user re-sees an already-dismissed notification once.
- **Process env mutation**: `process.env["deepnest_debug"]` is read each call to `getAppNotifications` ‚Äî flipping the env var at runtime would change behaviour mid-session. No code in this repo flips it; called out for awareness.

## 10. Other discoveries

- The class constructs `app.getVersion()` at construction time (`notification-service.js:9`). Calling `new NotificationService()` before `app.ready` is technically supported by Electron's `app.getVersion`, which works pre-ready, but the `seenDataPath` resolution depends on `app.getPath('userData')` which **also** works pre-ready since Electron 5. Today the constructor runs at module load (`main.js:66`), which is before `app.ready` ‚Äî verified safe.
- The merged-notification HTML embeds `<style="margin-bottom: 1em;">`-style inline attributes that duplicate the stylesheet in `notification.html`. Acceptable today; flagged if HTML refactoring happens.
- `formatFileSize` (`notification-service.js:360-368`) uses 1024-step units but labels them `KB / MB / GB / TB` (should be KiB/MiB/‚Ä¶). Cosmetic.
