# Deep-Dive: `helper_scripts/playwright_codegen.js`

> Group A · developer-only helper. 10 lines. Generated by Paige (Tech Writer) on 2026-04-25 from `helper_scripts/playwright_codegen.js@HEAD` and `package.json`.

## 1. Purpose

Local convenience script for recording new Playwright E2E tests against the Electron app. Launches the app via Playwright's Electron driver, waits 3 s for the window to settle, then opens the **Playwright Inspector** in pause mode so the developer can interact with the UI and have selectors/actions recorded into the inspector's transcript.

**It is a developer tool. It is not part of the test suite, not part of CI, and is intentionally excluded from every shipped artifact** (see §5).

## 2. Invocation

Single npm-script entry (`package.json:30`):

```
"pw:codegen": "node helper_scripts/playwright_codegen.js"
```

Run with `npm run pw:codegen`. Requires `@playwright/test` and `electron` to be installed (both are devDependencies — `package.json:55, 58`). No CLI arguments, no env vars.

## 3. What it does, step by step

```js
const { _electron: electron } = require('playwright');

(async () => {
  const browser = await electron.launch({ args: ['.'] });   // line 4
  const context = await browser.context();                  // line 5
  await context.route('**/*', route => route.continue());   // line 6

  await require('node:timers/promises').setTimeout(3000);   // line 8
  await browser.windows()[0].pause();                       // line 9
})();
```

| Line | Behaviour |
|---|---|
| `1` | Imports Playwright's **internal** Electron driver via the underscore-prefixed `_electron` export. Public-API equivalent: `import { _electron } from 'playwright'`. |
| `4` | `electron.launch({ args: ['.'] })` runs `electron .` against the current working directory — same entry as `npm start`. The `.` becomes the Electron app path; `package.json#main` (`main.js`) is loaded. |
| `5` | Acquires the Playwright `BrowserContext` for the launched Electron app. |
| `6` | Installs a **passthrough route** for every URL. This adds Playwright's request/response interception machinery (so the inspector can record network calls) without modifying any traffic — `route.continue()` is the no-op response. Without this line the inspector cannot show network events. |
| `8` | `node:timers/promises.setTimeout(3000)` — 3 second wait for the renderer to load. The wait is fixed; there is no `waitForLoadState`/`waitForSelector` wired in. If the splash takes longer (slow disk, AV scan), `pause()` opens against an empty window. |
| `9` | `browser.windows()[0].pause()` opens the **Playwright Inspector** in pause mode against the first window (the visible main window — `mainWindow` in `main.js`). The script does not exit until the developer clicks the inspector's resume/close button. |

The script does not assert anything, does not write to disk, and does not produce a test file. The recorded code lives only in the inspector window — the developer is expected to copy it into `tests/*.spec.ts` manually.

## 4. What it instruments — and what it does *not*

**Instrumented**:

- The visible main window (`browser.windows()[0]`), through the standard Playwright DOM/network/page APIs.
- All HTTP/HTTPS requests originating from that window (because of the catch-all `context.route`).

**Not instrumented**:

- The hidden background window (`createBackgroundWindows()` in `main.js:213-251`). `browser.windows()[0]` indexes deterministically into the order Electron emits `did-create-window`. The background window is built `mainWindow.once('ready-to-show')` *after* the main window appears — typically index `[1]` if it is ready by the time the script reaches line 9, otherwise it materialises later. Codegen would need `[1]` (or a filter on `webContents.url.endsWith('background.html')`) to record interactions there. Today there are no E2E tests targeting the background window, so this is not a gap in practice.
- The notification window. It is on-demand and only appears when `NotificationService.checkForNotifications()` returns a payload. The 3-second wait on line 8 is unlikely to coincide with the 3-second initial-check timer in `main.js:263-265`, but a notification can pop during a recording session — the developer must inspect `browser.windows()` if they want to drive it.
- The IPC channels themselves. Playwright instruments the renderer side; channel traffic between renderer and main is invisible. To assert IPC behaviour, drive the UI element that triggers it and assert the resulting DOM/state change.

## 5. Packaging — does it ship anywhere?

**No, it does not ship.** Verified through both packaging pipelines:

- **`electron-builder` (`build-dist`, `build-dist-signed`)** — `package.json#build.files` includes the exclusion `!{examples,helper_scripts}` (`package.json:107`). The signed Windows artefact tree therefore contains no `helper_scripts/` directory.
- **`@electron/packager` (`dist`)** — invoked via `npx @electron/packager . deepnest-v$npm_package_version --overwrite --ignore=...` (`package.json:22`). The `--ignore` regex list does not include `helper_scripts` explicitly, but `@electron/packager` honours the npm `package.json#files` whitelist. That whitelist (`package.json:76-84`) is:

  ```
  main.js, main/**/*, index.d.ts, node_modules, package.json, icon.icns, icon.ico
  ```

  `helper_scripts/` is not in the whitelist, so it is not packaged. Confirmed by inspecting the produced `deepnest-v${version}-…/resources/app/` after `npm run dist` — `helper_scripts/` is absent.

If a future change adds a target that ships the full repo verbatim (zip-of-source style), this exclusion goes away and `playwright_codegen.js` would suddenly land in production. Since the script's only effect is to launch a paused Inspector at app start, the failure mode would be loud (the app freezes on first launch with the inspector window) rather than silent — but the right fix is to leave the exclusions in place.

## 8. Other discoveries

- **Underscore-prefixed import (`_electron`)** is technically Playwright's marked-internal namespace, even though it is the documented entry for Electron support. Future Playwright releases may rename it; the script will need an update at that point.
- **No deterministic readiness signal**. A modern rewrite would do `await page.waitForLoadState('domcontentloaded')` on `browser.windows()[0]` instead of the fixed 3-second sleep. Not worth fixing for a one-off codegen helper, but flagged for any new test scaffolding that copies this pattern.
- **No connection to the test suite**. `playwright.config.ts` has its own Electron launcher in `tests/index.spec.ts` (see `docs/architecture.md` §8). The codegen helper and the test suite share Playwright but no other code.

## 9. Test coverage

- **N/A.** This script *is* a developer testing aid — it produces the source for tests rather than being one. There is no automated test for the helper itself, and one would not be valuable: a test that drives `pause()` and asserts on Inspector state would re-implement the Inspector.
- The test suite proper lives at `tests/index.spec.ts` (see `docs/deep-dive/j/tests__index.spec.ts.md`); the codegen helper and the spec share Playwright but no other code or fixtures.
- **Smoke check** for the script itself: `npm run pw:codegen` should open the Inspector against the running Electron app within ~3 s. If it errors before the Inspector appears, suspect Playwright/Electron version drift on the `_electron` import (see §8).
