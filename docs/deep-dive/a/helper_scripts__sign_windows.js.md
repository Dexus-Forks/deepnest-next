# Deep-Dive: `helper_scripts/sign_windows.js`

> Group A ¬∑ build-pipeline helper. 47 lines, exports a single `default` async function consumed by `electron-builder`. Generated by Paige (Tech Writer) on 2026-04-25 from `helper_scripts/sign_windows.js@HEAD` and `package.json#build`.

## 1. Purpose

Per-file Authenticode signing wrapper for the Windows release build. Wires Microsoft `signtool.exe` against the Certum hardware-backed code-signing certificate identified by SHA-1 thumbprint `e0cdd96f315959b8d333807585c4868f22d4f396`. Time-stamped via Certum (`http://time.certum.pl`), digest SHA-256.

## 2. Wiring (how it is invoked)

This script is **not** an `afterSign` hook ‚Äî it is the per-file `sign` callback registered in `electron-builder`'s Windows signtool options. From `package.json#build.win.signtoolOptions` (`package.json:135-140`):

```json
"signtoolOptions": {
  "sign": "./helper_scripts/sign_windows.js",
  "signingHashAlgorithms": ["sha256"]
}
```

`electron-builder` `require()`s the path at build time and calls the exported `default` async function once per artifact whose extension is in `package.json#build.win.signExts` (`package.json:126-134`):

```
.node, .exe, .dll, .sys, .msi, .appx, .appxbundle
```

`signAndEditExecutable: true` and `forceCodeSigning: true` (`package.json:124-125`) mean an unsigned Windows build is a build failure, not a warning.

The npm scripts that exercise this path are `build-dist` and `build-dist-signed` (`package.json:23-24`):

```
"build-dist":        "electron-builder build --win",
"build-dist-signed": "electron-builder build --win --publish never"
```

`dist-all-signed` chains `clean-all ‚Üí install ‚Üí build ‚Üí build-dist-signed` (`package.json:26`) for a full reproducible signed build.

The `npm run dist` script uses `@electron/packager` instead and **does not** invoke this signer ‚Äî `dist` produces an unsigned tree (`package.json:22`).

### 3.3 Skip-list mismatch (callout)

`package.json#build.win.signExts` declares **seven** signable extensions: `.node, .exe, .dll, .sys, .msi, .appx, .appxbundle`.

This script's allow-list (`sign_windows.js:11`) is **three**: `.node, .exe, .dll`.

Effect: any `.sys`, `.msi`, `.appx`, or `.appxbundle` artefact handed to this signer logs `Skipping non-EXE/DLL: ‚Ä¶` and **returns `true`** ‚Äî `electron-builder` treats it as signed and ships it unsigned. Today the build emits only `.dir` + `.zip` targets (`package.json:118-121`), neither of which produces those extensions, so this is latent. It becomes a real signing-bypass the moment someone adds `nsis`/`msi`/`appx` to `win.target`. Recommended fix: align the allow-list with `signExts`, or fail loudly on extensions outside the supported set.

| Certum SimplySign / proCertum CardManager middleware | Required | ‚Äî | Provides the cryptographic provider that `signtool` calls into. Without it, `signtool` errors with `SignerSign() failed.` (HRESULT varies). |
| Working network to `http://time.certum.pl` | Required | ‚Äî | RFC 3161 timestamp authority. Failure causes `signtool` to error and this script to return `false`. There is no fallback TSA. |

## 5. Failure modes that should NOT silently pass

The script's existing log lines surface the easy errors. These are the modes worth watching for in CI logs because they either kill the build correctly *or* succeed in an unsafe way:

| Symptom | Outcome | Severity |
|---|---|---|
| `Missing config.path` | `false` ‚Üí build fails | Correct, but unlikely (electron-builder always sends a path). |
| `signtool.exe not found at: ‚Ä¶` | `false` ‚Üí build fails | Correct. The hint about `SIGNTOOL_PATH` is the only doc; surface in any onboarding guide. |
| Skipping non-EXE/DLL: e.g. `.sys`, `.msi`, `.appx`, `.appxbundle` | `true` ‚Üí ships unsigned | **Silent pass.** See ¬ß3.3. Only safe today because the build does not emit those extensions; will silently break signing the moment a new target is added. |
| `signtool` exits non-zero (smart card unplugged, PIN cancelled, TSA timeout, expired cert) | `console.error('Signing failed:', err.message)` ‚Üí `false` ‚Üí build fails | Correct, but the message logged is `err.message`, not signtool's stderr. Because `stdio: 'inherit'` is set (`sign_windows.js:38`), the actual signtool output is interleaved with the build log ‚Äî check the lines immediately *before* `Signing failed:` for the real cause. |
| Wrong SDK installed (default `signtool` path missing) | `false` ‚Üí build fails | Same as missing-signtool. |
| Thumbprint typo / cert removed from store | signtool fails ‚Üí build fails | Correct, but the script does not pre-validate certificate availability ‚Äî every artefact triggers the same prompt loop before failing. |

There is no opportunity for a "succeeded but did not actually sign" outcome on the supported extensions: `signtool sign` either finishes with exit-0 (signed) or non-zero (raised). The risk is entirely the skip-list bypass in ¬ß3.3.

## 6. signtool command details

Built and `execSync`'d at `sign_windows.js:31-39`:

```
"<signtoolPath>" sign /sha1 e0cdd96f315959b8d333807585c4868f22d4f396 \
                       /tr http://time.certum.pl \
                       /td sha256 \
                       /fd sha256 \
                       /v /debug \
                       "<filePath>"
```

Flag-by-flag:

| Flag | Meaning |
|---|---|
| `/sha1 <thumbprint>` | Select the certificate by its SHA-1 thumbprint from the Windows cert store. |
| `/tr <url>` | RFC 3161 timestamp server (Certum). |
| `/td sha256` | Timestamp digest algorithm. |
| `/fd sha256` | File digest algorithm (matches `signingHashAlgorithms` in `package.json`). |
| `/v` | Verbose. |
| `/debug` | Print extra diagnostic info ‚Äî included intentionally to make CI logs forensically useful. |
| `<filePath>` | The artefact to sign. |

`execSync` is invoked with `{ stdio: 'inherit', shell: true }`. `shell: true` is what makes the Windows-style quoted command line parse correctly; it also means the command is interpreted by the system shell (`cmd.exe`). The `filePath` is interpolated directly into the command string and quoted with `"‚Ä¶"` ‚Äî this is the standard Windows convention but is **not robust** against paths containing literal `"` characters. Today no electron-builder artefact has those, but a hardened version should pass `[bin, ...args]` to a non-shell spawn instead of a string.

## 9. Other discoveries

- `nodeGypRebuild: false` in `package.json#build` (`package.json:88`) ‚Äî native addons are not rebuilt by electron-builder during the signed pipeline; the `postinstall` `electron-builder install-app-deps` is the only rebuild step. If a `.node` file ships into the pipeline pre-built against the wrong Electron ABI, signing it does not fix the ABI mismatch ‚Äî it just blesses a broken binary.
- `package.json#build.files` excludes `helper_scripts/` (`!{examples,helper_scripts}` at `package.json:107`). The signer is read by electron-builder before any artefact is produced; it is **not** packaged into the installer. Confirmed safe.
- `package.json#build.directories.output` is `deepnest-v${version}-win32-${arch}` (`package.json:115`). The output folder name does not include "signed" ‚Äî there is no on-disk way to distinguish a signed build from an unsigned one at the directory level. Use `signtool verify /pa /v <file>` to confirm.
