# Deep-Dive: `presets.js`

> Group A · privileged Node persistence module. 30 lines of code, three exported functions. Generated by Paige (Tech Writer) on 2026-04-25 from `presets.js@HEAD`.

## 1. Purpose

CRUD over a single JSON file at `app.getPath('userData')/presets.json`. Each top-level key is a preset name; each value is an opaque string (a JSON-serialised `UIConfig` produced by the renderer). Required by `main.js` and exposed to the renderer via three `ipcMain.handle` channels.

## 2. Public surface

```js
module.exports = { loadPresets, savePreset, deletePreset };
```

| Function | Signature | Returns | Throws |
|---|---|---|---|
| `loadPresets()` | `() => Record<string, string>` | parsed file contents, or `{}` if the file does not exist | propagates `JSON.parse` errors and `fs.readFileSync` IO errors |
| `savePreset(name, config)` | `(name: string, config: string) => void` | — | propagates `fs.writeFileSync` IO errors |
| `deletePreset(name)` | `(name: string) => void` | — | propagates `fs.writeFileSync` IO errors |

The `config` argument to `savePreset` is **already a JSON string** when called via the IPC layer — the renderer's `PresetService.savePreset` does `JSON.stringify(config)` before invoking the channel (`main/ui/services/preset.service.ts:140-148`). Inside `presets.js`, that string is treated as opaque and stored as-is under `presets[name]`.

## 3. Storage contract

- **Location**: `app.getPath('userData')/presets.json` (resolved once at module load — `presets.js:5`).
- **Encoding**: UTF-8 JSON. The file is read with the default `fs.readFileSync(path).toString()`.
- **Schema**: `{ [presetName: string]: string }` where each value is a stringified `UIConfig` (see `main/ui/types/index.ts` — `PresetConfig`).
- **Format**: pretty-printed with `JSON.stringify(presets, null, 2)` on every write.

Example file (truncated):

```json
{
  "Plywood Default": "{\"units\":\"mm\",\"scale\":72,\"spacing\":2,...,\"conversionServer\":\"https://converter.deepnest.app/convert\"}",
  "Plywood 6mm Tight": "{\"units\":\"mm\",\"scale\":72,\"spacing\":0.5,...}"
}
```

## 4. Flow

### 4.1 `loadPresets()` — `presets.js:7-12`

1. `fs.existsSync(presetsPath)`.
2. If present: `JSON.parse(fs.readFileSync(presetsPath).toString().replaceAll(...))` — see §5.
3. If absent: `return {}`.

There is no in-memory cache at this layer. Every call hits disk. (The renderer-side `PresetService` does cache the result and invalidates on save/delete — `main/ui/services/preset.service.ts:32-205`.)

### 4.2 `savePreset(name, config)` — `presets.js:14-18`

1. `loadPresets()` — full read + parse.
2. `presets[name] = config` — overwrites without warning if the key already exists.
3. `fs.writeFileSync(presetsPath, JSON.stringify(presets, null, 2))`.

### 4.3 `deletePreset(name)` — `presets.js:20-24`

Identical to `savePreset` except step 2 is `delete presets[name]`. Calling `deletePreset` with an unknown name is a no-op (the file is still rewritten with the unchanged contents — wasted IO but no error).

## 5. URL-rewrite invariant

`loadPresets` rewrites the legacy converter host inside the raw file body **before** parsing (`presets.js:9`):

```
http://convert.deepnest.io  → https://converter.deepnest.app/convert
https://convert.deepnest.io → https://converter.deepnest.app/convert
```

This is the same migration `main.js:355` performs on `settings.json`. Old presets that mention the legacy converter inside a stringified `UIConfig` value get migrated transparently on every load. `savePreset` and `deletePreset` round-trip whatever the renderer hands them — they do not re-apply the rewrite, but they also never need to (the renderer's `PresetService.migrateConversionServer` runs again on `getPreset`).

## 6. Atomicity guarantees — *or lack thereof*

This module makes **none of the standard write-safety guarantees**. Concretely:

| Guarantee | Provided? | Why it matters |
|---|---|---|
| Atomic write (temp-file + rename) | **No** — `fs.writeFileSync` writes in place. | A crash mid-write can truncate `presets.json`. |
| Lock against concurrent writes | **No** — no `flock`, no in-process queue. | Two near-simultaneous `save-preset` invocations from the renderer race in the read-modify-write window. The last writer wins; intervening edits silently disappear. |
| Read-after-write consistency | **No** — no cache invalidation event. | Renderer invalidates its own cache after `save-preset` succeeds; any other reader holds stale data until it calls `loadPresets()` again. |
| Schema validation | **No** — values are opaque strings. | A renderer bug that ships malformed JSON will surface only when `PresetService.getPreset` calls `JSON.parse` on read. |
| Backup / rollback | **No**. | No `.bak`. A bad write is not recoverable from this module. |

**Realistic blast radius today**: low. There is exactly one renderer process talking to this file via `ipcMain.handle`, the IPC handlers are awaited serially, and the user only writes presets via explicit save actions. The race window opens if a future feature does background writes (auto-save, sync, import).

**If this needs to change**: write to `presets.json.tmp`, `fs.fsyncSync` it, then `fs.renameSync` over `presets.json`. The same pattern would belong in `main.js`'s `write-config` handler.
