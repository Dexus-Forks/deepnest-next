# Deep-Dive: `main.js`

> Group A · privileged Node entry. Generated by Paige (Tech Writer) on 2026-04-25 from a line-by-line read of `main.js@HEAD`. Cross-references `docs/architecture.md` §2, §5.

## 1. Purpose

Electron app entry point. Responsibilities, in order of execution:

1. Acquire single-instance lock; quit if a second copy is launched.
2. Initialise `@electron/remote`, crash reporter, listener cap.
3. On `app.ready`: build the visible main window, then 1× hidden background window, then schedule the notification poller.
4. Host every `ipcMain.on` / `ipcMain.handle` handler — both the nesting fan-out and the persistence handlers.
5. Own the on-demand modal notification window.
6. On `before-quit`: wipe `./nfpcache/`.

Declared as the package entry via `package.json#main` (`main.js`).

### 3.2 `createBackgroundWindows()` — `main.js:213-251`

Recursive constructor that stops when the module-scope `winCount >= 1`. The comment _"used to have 8, now just 1 background window"_ (`main.js:217`) is the only in-source record of the historic worker count.

| Property | Value | Note |
|---|---|---|
| `show` | `false` always | hidden compute renderer |
| URL | `file://${__dirname}/main/background.html` | Tier B, native NFP addon |
| webPreferences | identical to main window | needs `nodeIntegration` to `require('@deepnest/calculate-nfp')` |
| busy flag | `back.isBusy` mutated on `background-start`/`background-response` | round-robin would need this if `winCount > 1` |
| Crash listeners | both `webContents.on('render-process-gone')` and `back.on('render-process-gone')` log to console | no restart logic |

### 3.3 `createNotificationWindow(notification)` — `main.js:148-203`

Created on demand; closed by user or by `close-notification` IPC. Important properties:

- `parent: mainWindow`, `modal: true`, `alwaysOnTop: true`, `type: 'notification'`.
- Fixed 750×500, non-resizable, non-min/maximisable.
- Same permissive `webPreferences` (no `nativeWindowOpen` — only main + background carry that).
- Notification payload is **stashed on the BrowserWindow instance** as `notificationWindow.notificationData = notification` (`main.js:202`). The renderer fetches it later via the `get-notification-data` IPC.
- On close, `notificationWindow = null`. `closed` is the only place the reference is cleared — `close-notification` re-uses `BrowserWindow.fromWebContents(event.sender)` and invokes `win.close()`.

If a second notification arrives while a window is open, `createNotificationWindow` calls `.close()` on the previous one before opening the next (`main.js:149-151`).

## 4. App lifecycle wiring

| Event | Handler | What it does |
|---|---|---|
| `app.requestSingleInstanceLock()` (`main.js:69-80`) | quit if not acquired; otherwise on `second-instance` focus the existing main window | only one running copy of the app |
| `app.whenReady()` (`main.js:83-86`) | empty body (`//myWindow = createWindow()`) — actual creation lives in the second `app.on('ready')` below | dead branch; safe to remove eventually |
| `app.on('ready')` (`main.js:256-274`) | `createMainWindow()` → on `ready-to-show`: `mainWindow.show()`, `createBackgroundWindows()`, schedule notifications (3 s, then every 30 min) | ordering matters: bg windows are not built until the visible window has shown |
| `app.on('window-all-closed')` (`main.js:277-279`) | `app.quit()` | no macOS-specific keep-running |
| `app.on('activate')` (`main.js:281-287`) | recreate main window if `mainWindow === null` | macOS dock-click case |
| `app.on('before-quit')` (`main.js:289-297`) | sync-deletes every file in `./nfpcache/` if it exists | scratch directory cleanup; does **not** clean `global.NEST_DIRECTORY` |
| `mainWindow.on('closed')` (`main.js:271-273`) | `app.quit()` | redundant with `window-all-closed` on this platform set, but explicit |
| `app.on('render-process-gone')` (`main.js:11`) | log the failure | no restart |

## 6. Security invariants

`contextIsolation: false` + `nodeIntegration: true` + `enableRemoteModule: true` apply to:

- **Main window** (`main.js:100-105`) — required because `main/ui/index.ts` does `require('electron')`, `require('graceful-fs')`, `require('@deepnest/svg-preprocessor')`, etc. Removing `nodeIntegration` would force a preload-bridge rewrite of every service in `main/ui/services/`.
- **Background window** (`main.js:221-226`) — required because `main/background.js` does `require('@deepnest/calculate-nfp')` (native addon). Removing `nodeIntegration` would require relocating the addon load to a preload script — non-trivial because the addon expects an Electron renderer ABI.
- **Notification window** (`main.js:167-171`) — required because `main/notification.html:199` does `const { ipcRenderer } = require("electron")`. Removing `nodeIntegration` here would require a preload script that exposes `ipcRenderer.send` / `ipcRenderer.on`. This is the most tractable hardening target — the notification window does not need the rest of Node.

The compensating control is `setWindowOpenHandler` on both main and notification windows, which routes any `window.open(url)` to `shell.openExternal` and denies the in-app pop-up. There is **no** equivalent guard on the background window (it never tries to open URLs by design).

`crashReporter.start({ uploadToServer: false })` (`main.js:16`) keeps minidumps local; `console.log(crashReporter.getLastCrashReport())` runs at startup to surface the previous crash to whoever started the app from a terminal.

## 7. URL-rewrite invariant

Both the config handler (`main.js:355`) and `presets.js:9` perform the same string replacement:

```
http://convert.deepnest.io  → https://converter.deepnest.app/convert
https://convert.deepnest.io → https://converter.deepnest.app/convert
```

The DXF→SVG converter URL was renamed; old user data is migrated transparently on read. Any new code that reads either file must apply the same migration (or call through these handlers / `presets.js`).

The renderer-side `PresetService.migrateConversionServer` (`main/ui/services/preset.service.ts:52-59`) performs the same migration when **parsing** an individual preset value. The two layers are belt-and-braces — both will run on every read.

## 9. Other discoveries

- `main.js:144` uses `fs.mkdirSync` without `{ recursive: true }`. `os.tmpdir()/nest` works because `os.tmpdir()` is guaranteed to exist; if `SAVE_PLACEMENTS_PATH` points at a non-existent multi-level directory the call will throw. Not a regression today — flagged for any change that allows arbitrary user paths.
- `main.js:289-297` deletes files in `./nfpcache/` (CWD-relative), not `userData`. If the app is launched from a write-protected CWD this will throw on quit; today the dir does not exist outside dev workflows so the `existsSync` short-circuits.
- The `renderer-process-gone` log on the main window object (`main.js:11`) and on each background window (`main.js:248-249`) **do not** rebuild the failed window. A bg-window crash leaves `backgroundWindows[i]` pointing at a destroyed object until the next `background-stop` rebuilds them.
- `runNotificationCheck()` (`main.js:205-210`) is invoked in three places: 3 s after main window shows, every 30 min thereafter (`main.js:267-269`), and after the user dismisses a notification (`main.js:413-418`). Wake-from-sleep + clock skew is not handled — the 30-min interval keeps drifting on macOS App Nap.

## 10. Test coverage

- **Implicitly exercised by the only Playwright spec.** `tests/index.spec.ts` (see `docs/deep-dive/j/tests__index.spec.ts.md`) launches the app via Playwright's Electron driver with `main.js` as `package.json#main`, so the entire startup path — single-instance lock, `createMainWindow`, `createBackgroundWindows`, the `background-start`/`background-response`/`background-progress` round-trip — runs as part of the nesting smoke test.
- **No direct assertion** on any IPC channel listed in §5. The spec asserts on renderer state (`window.DeepNest.nests`); the channels are covered by virtue of "if the round-trip were broken, the assertion would never satisfy."
- **Not covered**: notification window flow (the spec finishes before the 3 s initial poll fires reliably; even if it did, the test does not drive `close-notification`), preset CRUD (`load-presets`/`save-preset`/`delete-preset`), `read-config`/`write-config`, `login-success`/`purchase-success`, `setPlacements`/`test`, `before-quit` cache wipe, `setWindowOpenHandler`, single-instance lock loss, and `render-process-gone` paths.
- **No unit tests** exist for any handler in this file; everything is integration-only via the Electron launch.
