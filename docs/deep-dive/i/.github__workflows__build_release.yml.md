# Deep-Dive: `.github/workflows/build_release.yml`

> Group I · CI / release. 233 lines, three jobs (Linux / macOS / Windows), 5 matrix cells total. Generated by Paige (Tech Writer) on 2026-04-26 from `.github/workflows/build_release.yml@HEAD` for [DEE-41](/DEE/issues/DEE-41) (parent: [DEE-11](/DEE/issues/DEE-11)).

## 1. Filename discrepancy (DEE-11 callout)

The DEE-11 parent issue scope listed this file as `.github/workflows/release.yml`. The actual filename in the repo is **`build_release.yml`**. Per DEE-11 / DEE-41 instructions: flagging the discrepancy here. The workflow `name:` field is `build release` (with a space), which is what shows up in the GitHub Actions UI and in the required-status-check picker — search the branch-protection settings under that label, not under the filename.

## 2. Purpose

Builds, packages, and uploads the per-platform Electron Packager artifacts to a **published GitHub Release**. Triggered by the `release` event (type `published`). The job does **not** create the release — it expects a release already to exist (drafted and published manually or via the GitHub UI / `gh release create`) and looks it up via `github.rest.repos.getLatestRelease`.

## 3. Triggers

| Trigger | Filter | Notes |
|---|---|---|
| `workflow_dispatch` | _none_ | Manual run. Dangerous: it still queries `getLatestRelease` and will attach assets to whatever release is currently latest. |
| `release` | `types: [published]` | Fires once per release transition into "published". Does **not** fire on draft / pre-release transitions. |

There is **no `repository_dispatch`** and no tag-based filter. A push of a tag does not trigger this workflow; the release event does.

## 5. Step-by-step (representative — Linux job)

### 5.1 Checkout

```yaml
- uses: actions/checkout@v4
  with:
    submodules: recursive
```

Same `submodules: recursive` no-op as in [`build.yml`](./.github__workflows__build.yml.md) §4.1 — kept defensively, no current submodule registrations.

### 5.2 Set architecture

```yaml
- name: Set architecture
  id: set_arch
  run: |
    if [[ "${{ matrix.os }}" == "ubuntu-24.04-arm" ]]; then
      echo "ARCH=arm64" >> $GITHUB_ENV
    else
      echo "ARCH=x64" >> $GITHUB_ENV
    fi
```

ARCH is exported through `$GITHUB_ENV` so subsequent `run:` blocks see `${ARCH}`. Bash on Linux/macOS, PowerShell on Windows (`echo "ARCH=x64" >> $env:GITHUB_ENV`).

### 5.3 Get latest release

```yaml
- uses: actions/github-script@v6
  with:
    script: |
      const release = await github.rest.repos.getLatestRelease({
        owner: context.repo.owner,
        repo: context.repo.repo
      });
      core.setOutput("upload_url", release.data.upload_url);
      core.setOutput("release_id", release.data.id);
      core.setOutput("tag", release.data.tag_name);
```

Outputs `upload_url` and `tag` are consumed by the next two steps. Implicitly trusts that the release that just fired the trigger **is** the latest release. On `workflow_dispatch` from a busy maintainer with multiple drafts, this assumption can attach the wrong assets — see §6 _Manual-dispatch race_.

### 5.4 Build

```yaml
- name: Build project
  run: |
    npm install
    npm run build
```

`npm install` (not `npm ci`) — same property as [`build.yml`](./.github__workflows__build.yml.md). `npm run build` runs `tsc && electron-builder install-app-deps` (`package.json:19`).

### 5.5 Package

```yaml
- name: Package build
  run: |
    TAG=${{ steps.get_release.outputs.tag }}
    BUILD_DIR="deepnest-${TAG}-${BUILD_FOR_OS}-${ARCH}"
    npx @electron/packager . "deepnest-${TAG}" --overwrite
    zip -r "${BUILD_DIR}.zip" "$BUILD_DIR"
```

The packager output directory name comes from `@electron/packager` defaults, so the actual on-disk folder name will be `deepnest-${TAG}-${platform}-${arch}` — which the `zip` command then bundles. Windows uses `7z a` instead of `zip` (line 222).

### 5.6 Upload release asset

```yaml
- uses: actions/upload-release-asset@v1
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  with:
    upload_url: ${{ steps.get_release.outputs.upload_url }}
    asset_path: deepnest-${{ steps.get_release.outputs.tag }}-${{ env.BUILD_FOR_OS }}-${{ env.ARCH }}.zip
    asset_name: deepnest-${{ steps.get_release.outputs.tag }}-${{ env.BUILD_FOR_OS }}-${{ env.ARCH }}.zip
    asset_content_type: application/zip
```

`actions/upload-release-asset@v1` is **archived / unmaintained** (the canonical replacement is `softprops/action-gh-release` or the `gh` CLI). It still works but is a known migration target. The step name string `Artefakt an letztes Release anhängen` is German — no functional impact, but worth normalising for repo-wide search consistency.

## 6. Codesign hand-off — the critical Group A cross-link

**The Windows release path produced by this workflow is unsigned.**

The DEE-11 / DEE-41 brief flagged "the codesign hand-off to [`helper_scripts/sign_windows.js`](../a/helper_scripts__sign_windows.js.md) (Group A) is the most important cross-link." The hand-off, as wired today, is **broken at the workflow level**:

| Layer | Configured? | Actually executed by this workflow? |
|---|---|---|
| `package.json#build.win.signtoolOptions.sign = ./helper_scripts/sign_windows.js` (`package.json:135-140`) | yes | **no** |
| `forceCodeSigning: true`, `signAndEditExecutable: true` (`package.json:124-125`) | yes | **no** |
| `helper_scripts/sign_windows.js` ([Group A deep-dive](../a/helper_scripts__sign_windows.js.md)) — Authenticode wrapper around `signtool.exe` | implemented | **never invoked** |
| `electron-builder build --win` (`scripts.build-dist`, `scripts.build-dist-signed` in `package.json:23-24`) — the **only** scripts that exercise the signer | defined | **not called from any workflow** |
| `npx @electron/packager` (this workflow's `build-and-release-windows` job, line 220) | yes | yes — but `@electron/packager` does not invoke `electron-builder`'s `signtoolOptions` |

Group A's deep-dive has the same finding from the script side: see [`helper_scripts__sign_windows.js.md` §2 _Wiring_](../a/helper_scripts__sign_windows.js.md). The mismatch sits at the workflow level: this workflow runs `@electron/packager`, but the codesign hook is registered against `electron-builder`.

If signed Windows builds are required for release, one of two changes is needed:

1. Replace `npx @electron/packager` in `build-and-release-windows` with `npx electron-builder build --win --publish never` (or `npm run build-dist-signed`), and let `electron-builder` consume the `signtoolOptions` block from `package.json`.
2. Add an explicit `signtool.exe sign …` step in the Windows job that mirrors what [`helper_scripts/sign_windows.js`](../a/helper_scripts__sign_windows.js.md) does, plus the certificate / thumbprint / timestamp authority from that file.

Until then, the `signtoolOptions` block in `package.json` is **dead config** for the actual release path that ships to users.

## 7. Secrets and permissions

| Secret | Used in | Notes |
|---|---|---|
| `secrets.GITHUB_TOKEN` | `actions/upload-release-asset@v1` env (linux line 78, macOS line 157, windows line 228) | Default-scoped token. Needs `contents: write` to attach assets to a release; that permission is granted by default for the `release` trigger context but **must be added explicitly** if repo-level default permissions are tightened to read-only. |

No code-signing secrets are referenced. There is no Apple Developer ID notarisation, no Microsoft Authenticode certificate, and no `secrets.WINDOWS_PFX_PASSWORD`-style entry. The macOS job therefore produces unsigned, un-notarised `.app` bundles — Gatekeeper will block them on download.

## 8. Coverage gaps and smells

- **Codesign is dead config.** See §6.
- **macOS unsigned + un-notarised.** No `osxSign` / `osxNotarize` step; users get the "damaged, can't be opened" Gatekeeper dialog.
- **`actions/upload-release-asset@v1` is archived.** Migrate to `softprops/action-gh-release@v2` or `gh release upload`.
- **`getLatestRelease` race.** On `workflow_dispatch` the workflow attaches to whichever release happens to be latest, not to a release the operator selected. There is no `release_id` or tag input.
- **`npm install` not `npm ci`.** Lockfile drift in CI. Same finding as [`build.yml`](./.github__workflows__build.yml.md) §6.
- **No SHA-256 / signature publication of release artifacts.** Users have no way to verify a downloaded zip beyond the GitHub-served URL.
- **`fail-fast` default (`true`).** A Linux x64 failure cancels Linux arm64 mid-build, costing visibility on per-arch regressions.
- **Boost step is conditional on `startsWith(matrix.os, 'ubuntu')`** but lives inside the macOS job too (lines 102-103). It is dead code there — kept consistent with the Linux job, but the `if:` makes it skip cleanly. Remove for clarity.
- **macOS ARCH detection is wrong on `macos-13`.** Lines 110-118 set `ARCH=arm64` only when `matrix.os == "macos-latest"` and `ARCH=x64` otherwise. `macos-13` is x64 — that is correct. But once GitHub renames `macos-latest` (Apple Silicon), this string match will silently start mis-tagging. Use the runner's `runner.arch` (or `process.arch`) instead.
- **No `cache: npm`.** Same finding as [`build.yml`](./.github__workflows__build.yml.md) §6.
- **`asset_name` matches `asset_path`.** Trivial but means a future rename of the on-disk filename would also rename the user-visible download — coordinate any change.
