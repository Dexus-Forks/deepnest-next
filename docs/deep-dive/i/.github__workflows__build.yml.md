# Deep-Dive: `.github/workflows/build.yml`

> Group I · CI / release. 45 lines, one job (`build`), matrix of 4 cells (Node 20.x / 22.x × Windows / Ubuntu). Generated by Paige (Tech Writer) on 2026-04-25 from `.github/workflows/build.yml@HEAD`.

## 1. Purpose

Continuous-integration build for every push to `main` and every pull request. Verifies that the TypeScript / Electron-native stack still compiles and packages on two Node majors and two operating systems. **Does not produce or upload artifacts** — this workflow is a green-light gate, not a release pipeline. Release artifacts come from [`build_release.yml`](./.github__workflows__build_release.yml.md).

## 2. Triggers

| Trigger | Filter | Notes |
|---|---|---|
| `workflow_dispatch` | _none_ | Manual run from the Actions tab. |
| `push` | `branches: ["main"]` | Only `main` push fires this job. Feature-branch pushes do **not** trigger CI directly — they only run via PR. |
| `pull_request` | _no branch filter_ (commented out, `# branches: [ "main" ]`) | Runs against PRs targeting any base branch. |

There is **no path filter**. Doc-only changes, `.md` edits, or even `.gitignore` tweaks all fire the full matrix. Compare with [`playwright.yml`](./.github__workflows__playwright.yml.md), which restricts to code/config paths.

## 3. Job table

| Job | Runner | Steps | Secrets read | Artifacts produced |
|---|---|---|---|---|
| `build` | matrix: `windows-2022`, `ubuntu-22.04` × Node `20.x`, `22.x` (4 cells) | checkout (with submodules) → install Boost (Ubuntu only) → setup Node → `npm install && npm run build` → `npm run dist` | _none_ | _none_ (no `actions/upload-artifact` step) |

Total cells: **4**. There is no `fail-fast: false`, so the matrix uses the GitHub default (`true`) — the first cell that fails cancels the rest. That is the desired behaviour for a green-light gate but it can hide platform-specific regressions until the first-failing cell is fixed.

## 4. Step-by-step

### 4.1 Checkout (`actions/checkout@v4`)

```yaml
- uses: actions/checkout@v4
  with:
    submodules: recursive
```

`submodules: recursive` is set even though the repo currently has **no `.gitmodules`** at HEAD. This is a leftover safeguard for historical native deps; it is a no-op today but still costs a `git submodule` shell-out per cell.

### 4.2 Boost (Ubuntu only)

```yaml
- name: boost
  if: startsWith(matrix.os,'ubuntu')
  run: sudo apt-get update && sudo apt-get install -yq libboost-dev
```

`libboost-dev` is required by the native NFP / clipper deps under `node_modules/@deepnest/calculate-nfp` (see [`package.json:68`](../../../package.json)) at `npm install` time. On Windows the matching toolchain ships with the runner image; on macOS this CI workflow does not run at all (see §6 _Coverage gaps_).

### 4.3 Node setup

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: ${{ matrix.node-version }}
```

No `cache: npm`. `npm install` re-downloads the full dependency graph on every run. The release workflow has the same property — see [`build_release.yml`](./.github__workflows__build_release.yml.md) §5.

### 4.4 Build

```yaml
- name: Build
  run: |
    npm install
    npm run build
```

- `npm install` — **not** `npm ci`. Lockfile drift is therefore tolerated in CI; that is inconsistent with [`playwright.yml`](./.github__workflows__playwright.yml.md), which uses `npm ci`. See §6 _Lockfile inconsistency_.
- `npm run build` — runs `tsc && electron-builder install-app-deps` (`package.json:19`). The `install-app-deps` half is what triggers the native rebuild that needs Boost on Linux.

### 4.5 Package

```yaml
- name: Package
  run: |
    npm run dist
```

`npm run dist` invokes `@electron/packager` (`package.json:22`). The output tree is left in the workspace and **not uploaded** — there is no `actions/upload-artifact` step. The package only proves "it builds and packages without throwing"; downstream sanity checks (does it launch? are placements computed?) live in [`playwright.yml`](./.github__workflows__playwright.yml.md).

## 6. Coverage gaps and smells

- **macOS not covered.** The CI matrix is Linux + Windows only. The first time macOS is exercised is when `build_release.yml` runs on a published release, which is too late to catch a regression. See [`build_release.yml`](./.github__workflows__build_release.yml.md) §3 for the macOS matrix that is exercised at release time.
- **Lockfile inconsistency.** `build.yml` and `build_release.yml` use `npm install`; [`playwright.yml`](./.github__workflows__playwright.yml.md) uses `npm ci`. The CI build can therefore boot a different dependency tree than the test workflow — a regression in a transitive dep that breaks runtime tests will not be caught here.
- **No path filter on triggers.** Doc-only PRs (e.g. anything in `docs/deep-dive/`) burn 4 runner-minutes per push without testing anything that changed.
- **No `cache: npm` on `actions/setup-node`.** Every cell pays the full registry-download cost.
- **No artifact upload.** A breaking change that only surfaces at packaging time leaves no diagnostic blob to download.
- **`fail-fast` left default (`true`).** Hides cross-platform divergence: a Windows-Node-20 failure cancels Linux-Node-22 before its result is known.
- **Submodule recursion is a leftover.** No `.gitmodules` at HEAD; the flag is a no-op today.
- **`dist` step does not exercise signing.** `npm run dist` calls `@electron/packager`, which does not invoke [`helper_scripts/sign_windows.js`](../a/helper_scripts__sign_windows.js.md). Code-signing regressions are only caught by the `electron-builder`-based `build-dist` / `build-dist-signed` scripts (`package.json:23-24`), which are **not** wired into any workflow.
