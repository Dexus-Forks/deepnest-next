# Deep-Dive: `.github/workflows/playwright.yml`

> Group I · CI / release. 48 lines, one job (`test`), one matrix cell (`ubuntu-latest`). Generated by Paige (Tech Writer) on 2026-04-25 from `.github/workflows/playwright.yml@HEAD`.

## 1. Purpose

End-to-end runtime smoke test. Spins the packaged Electron app under `xvfb-run` on a Linux runner with a Playwright-driven Chromium install and uploads the HTML test report. This is the only workflow that proves the binary actually launches and that the renderer wires up — the [`build.yml`](./.github__workflows__build.yml.md) workflow only proves it compiles and packages.

## 2. Triggers

| Trigger | Filter | Notes |
|---|---|---|
| `workflow_dispatch` | _none_ | Manual run from the Actions tab. |
| `push` | `branches: [main]` + path filter (see below) | Runs on push to `main` only. |
| `pull_request` | `branches: [main]` + path filter | Runs on PRs targeting `main`. |

Path filter (applied to both `push` and `pull_request`):

```yaml
paths: ["**.json", "**.js", "**.ts", "**.jsx", "**.tsx", "**.yml"]
```

Doc-only changes (`*.md`, `*.txt`, anything under `docs/`) **do not** trigger this workflow. Compare with [`build.yml`](./.github__workflows__build.yml.md), which has no path filter and burns runner-minutes on doc PRs.

The path filter implicitly assumes that **no native source files** (`.cpp`, `.c`, `.h`, `.gyp`, `binding.gyp`) are part of the runtime path. Today's `node_modules/@deepnest/calculate-nfp` is shipped as a pre-built native dep, so the assumption holds; it would not hold if the repo started checking in C++ source again.

## 3. Job table

| Job | Runner | Steps | Secrets read | Artifacts produced |
|---|---|---|---|---|
| `test` | matrix: `ubuntu-latest` (one cell) | install Boost → checkout → setup Node `lts/*` → `npm ci` → fix `chrome-sandbox` permissions → `npm run build` → install Playwright Chromium → `xvfb-run npm test` (`DEBUG=pw:browser*`) → `upload-artifact: playwright-report/` | _none_ | `playwright-report/` (uploaded as `playwright-report`, retention 7 days) |

Total cells: **1**. There is no Windows / macOS Playwright run — Linux is the only platform exercised at the runtime level in CI.

`timeout-minutes: 5` (line 15) caps the entire job. A run that gets stuck (xvfb not starting, Chromium hanging, Electron busy-loop) will fail with a timeout rather than hold a runner indefinitely.

## 4. Step-by-step

### 4.1 Boost (before checkout — order matters)

```yaml
- name: boost
  if: startsWith(matrix.os, 'ubuntu')
  run: sudo apt-get update && sudo apt-get install -yq libboost-dev
```

`libboost-dev` is the same dep [`build.yml`](./.github__workflows__build.yml.md) §4.2 installs. Note the step runs **before `actions/checkout`** — that is fine because `apt-get` does not need the repo on disk. The `if:` is technically redundant for a single-cell `ubuntu-latest` matrix, but is kept consistent with the other workflows.

### 4.2 Checkout (no submodules)

```yaml
- uses: actions/checkout@v4
```

**`submodules: recursive` is NOT set here**, unlike [`build.yml`](./.github__workflows__build.yml.md) and [`build_release.yml`](./.github__workflows__build_release.yml.md) which both opt in. The repo currently has no `.gitmodules`, so this divergence is harmless today — but if a submodule is ever re-introduced, this workflow will break in a way the other two will not.

### 4.3 Node setup

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: lts/*
```

Uses the **floating LTS alias**, not a pinned major. The other two workflows pin to a matrix of `20.x` / `22.x` (`build.yml`) or `22.x` (`build_release.yml`). When Node bumps to a new LTS this job will pick it up automatically — that is normally a feature, but it means a Node-LTS-bump regression first appears in this workflow rather than in `build.yml`.

### 4.4 Install dependencies (`npm ci`)

```yaml
- run: |
    # npm install
    npm ci
```

Uses **`npm ci`**, not `npm install`. This is inconsistent with [`build.yml`](./.github__workflows__build.yml.md) (`npm install`) and [`build_release.yml`](./.github__workflows__build_release.yml.md) (`npm install`). A lockfile-drift bug that's invisible to the build workflows can surface here as `npm ci` fails the run.

The commented-out `# npm install` is a leftover; remove for clarity.

### 4.5 `chrome-sandbox` permission fix

```yaml
- name: Fix Ubuntu 24.04 issue https://github.com/electron/electron/issues/17972
  run: |
    file=node_modules/electron/dist/chrome-sandbox
    sudo chown root $file
    sudo chmod 4755 $file
```

Workaround for [electron/electron#17972](https://github.com/electron/electron/issues/17972) — the unprivileged user-namespace setup on Ubuntu 24.04 (and now `ubuntu-latest`) requires the chrome-sandbox SUID-root binary. Without this step, Electron fails to launch with `SUID sandbox helper binary was found, but is not configured correctly`.

### 4.6 Build

```yaml
- run: npm run build
```

`npm run build` runs `tsc && electron-builder install-app-deps` (`package.json:19`). `install-app-deps` is what triggers the Boost-dependent native rebuild — that's why `libboost-dev` had to be installed in step 4.1.

### 4.7 Install Playwright Chromium

```yaml
- run: npx playwright install --with-deps chromium
```

`--with-deps` installs the Chromium runtime dependencies via `apt-get`. Only Chromium is installed (no Firefox / WebKit) — the Playwright config exercises one browser project. Confirm in `playwright.config.ts` that no other projects are referenced; if they are, this step needs `firefox webkit` added.

### 4.8 Run the tests

```yaml
- run: xvfb-run npm test
  env:
    DEBUG: pw:browser*
```

- `xvfb-run` provides a virtual X display so Electron's renderer process can open a window on the headless runner.
- `npm test` resolves to `playwright test` (`package.json:18`).
- `DEBUG=pw:browser*` enables Playwright's browser-process namespace logs. They appear in the job's stdout, *not* in the uploaded report.

### 4.9 Upload report

```yaml
- uses: actions/upload-artifact@v4
  if: ${{ !cancelled() }}
  with:
    name: playwright-report
    path: playwright-report/
    retention-days: 7
```

`if: ${{ !cancelled() }}` uploads the report on success **and** on failure — only a manual cancel skips the upload. Retention is 7 days (default is 90), which keeps storage low; failed-run reports older than a week are gone.

## 6. Coverage gaps and smells

- **Linux-only.** Electron-on-Windows and Electron-on-macOS regressions are not detected by Playwright in CI. The Windows-specific signing / IPC paths (anything in [`helper_scripts/sign_windows.js`](../a/helper_scripts__sign_windows.js.md), `main.js` Windows branches) are completely untested at runtime.
- **Floating Node LTS.** `lts/*` is convenient but couples test stability to whatever LTS GitHub ships at run time. Pin to `22.x` (matching the release workflow) for reproducibility.
- **Path filter is brittle.** `**.yml` matches every YAML file in the tree; `**.json` matches `package-lock.json` (good — lockfile bumps re-run tests) but also matches every fixture JSON anywhere in the repo. A change to e.g. `tests/fixtures/legacy.json` re-runs the full E2E. Consider scoping to top-level paths (`src/**`, `main/**`, `package*.json`, `playwright.config.ts`).
- **No submodule recursion.** Diverges from the other two workflows. See §4.2.
- **`npm ci` vs other workflows' `npm install`.** Inconsistent — see [`build.yml`](./.github__workflows__build.yml.md) §6. Pick one across all three workflows.
- **Commented-out `# npm install` in step 4.4.** Dead comment, remove.
- **`timeout-minutes: 5` is tight.** A cold-cache `npm ci` plus `npm run build` plus `playwright install` can already take ~3 min. A flaky network or a slower runner image push the whole job past 5 min and trigger a false-fail timeout. Consider 10-15 min.
- **No `cache: npm` and no Playwright browser cache.** Every run downloads Chromium afresh.
- **Single `DEBUG` namespace (`pw:browser*`).** When a renderer hang occurs, `pw:browser*` shows the browser side but not the test framework side. Toggle between `pw:api*` (test-side) and `pw:browser*` (browser-side) via repo variable for easier diagnosis.
- **No JUnit / SARIF upload.** Only the HTML report is uploaded. PR check pages cannot show inline test failures without a JUnit reporter. Consider adding `--reporter=html,github` to surface failures in the GitHub UI.
